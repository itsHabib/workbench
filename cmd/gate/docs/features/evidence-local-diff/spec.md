# Evidence: local merge-base diff fallback for oversized PRs тАФ design spec

## Problem

`evidence.Gather` reads a PR's diff through `gh pr diff`, which calls the
GitHub `pulls/<n>` diff endpoint. That endpoint rejects diffs over 20,000
lines with HTTP 406 (`PullRequest.diff too_large`), so gate cannot gather
evidence for large PRs at all тАФ the run aborts before a verdict exists
(exit 4). Repo migrations hit this immediately: workbench#51 (the triage
move) is ~48k diff lines, and the coming gate move will be comparable.
A gate that cannot evaluate exactly the PRs that most need a floor
classification is a hole in the verification plane.

## Behavior

When тАФ and only when тАФ `gh pr diff` fails with the too-large rejection,
the diff evidence is computed locally from the same source of truth:

1. `gh api repos/<repo>/pulls/<n>` тЖТ the PR's `base.sha` + `head.sha`.
2. **Head pin**: the fetched `head.sha` must equal the `headRefOid` the
   run's view evidence already recorded. Mismatch = the PR moved between
   reads тЖТ fail, no fallback diff.
3. `gh api repos/<repo>/compare/<base>...<head>` тЖТ `merge_base_commit.sha`
   (metadata is returned even when the compare's file list is truncated).
4. Sanity gates before any git work: `mb` and `head` must be hex object
   names (so an API-sourced value can never reach git as an option), and
   `mb != head` (an equal base means the base advanced to contain head тАФ
   e.g. a concurrent merge тАФ which would yield an empty diff for a PR the
   406 just proved is over-cap; refuse rather than record it).
5. In a fresh temp dir: `git init`; `git fetch --depth=1 --end-of-options`
   of exactly the merge-base and head commits from
   `https://github.com/<repo>.git`; `git diff <mb> <head>`. Every git call
   runs with ambient configuration neutralized тАФ `GIT_CONFIG_SYSTEM` and
   `GIT_CONFIG_GLOBAL` point at the null device, `GIT_ATTR_NOSYSTEM=1` and
   `-c core.attributesFile=<null>` suppress the system and global/XDG
   *attributes* files (config nulling alone does not тАФ attributes files are
   read independently), all inherited `GIT_*` env is dropped, and the
   credential-helper list is cleared then set to `gh auth git-credential` тАФ
   so only explicit flags apply. The diff runs
   `--no-ext-diff --no-textconv --no-color` and `-c attr.tree=<head>`, so host
   textconv/format settings can't rewrite it and only the PR's *committed*
   `.gitattributes` are honored тАФ exactly as GitHub's server-side diff does.
6. An empty diff is refused (the 406 guaranteed a non-empty merge-base
   diff). The diff is recorded as the same `diffBody` evidence artifact,
   now stamped with `method`, `head`, and `merge_base` for audit.

Any failure at any step fails the run closed, exactly as a failed
`gh pr diff` does today. Errors other than the 406 too-large rejection
never trigger the fallback тАФ an auth or network failure still aborts.

## Why this is equivalent evidence

`gh pr diff` serves the three-dot (merge-base) diff, generated by git on
GitHub's side. `git diff <merge-base> <head>` over the same two commits,
with ambient config neutralized and committed attributes honored, is the
same computation; verified byte-identical against `gh pr diff` on
workbench#50 (599 lines), and produces the 47,945-line diff for
workbench#51 that the API refuses. Two depth-1 commit fetches carry no
history тАФ cost is proportional to the two trees, not the repo.

The `mb` (merge base) is sourced from the same authenticated GitHub API as
`head`, so it inherits тАФ does not widen тАФ the transport trust the whole
evidence chain (`gh pr view`, the CI rollup, the eventual merge) already
places in that API. Rename-detection budget can differ from GitHub's on a
very large migration, which can only *inflate* the diff (renames shown as
add+delete), so the floor over-classifies rather than under тАФ it fails
safe. Ancestry of `mb` is not asserted: a depth-1 shallow head cannot walk
back to the merge base, so `git merge-base --is-ancestor` is unavailable
here; hex-validation guards the argv, and the transport trust above covers
the value.

## Security posture

- **Same source, same auth.** The commits come from github.com over the
  gh-authenticated credential path. No local checkout, working tree, or
  operator-writable path participates; the temp dir starts empty and is
  removed after.
- **No ambient git config or attributes.** Every git call runs with
  `GIT_CONFIG_SYSTEM`/`GIT_CONFIG_GLOBAL` at the null device, inherited
  `GIT_*` env stripped, and the credential-helper list cleared before `gh`'s
  is set тАФ so a host `~/.gitconfig`, `GIT_EXTERNAL_DIFF`, `diff.noprefix`, or
  a rogue `url.insteadOf` cannot rewrite, reformat, or reroute the diff. The
  attributes files are read independently of config, so `GIT_ATTR_NOSYSTEM=1`
  and `core.attributesFile=<null>` additionally suppress the system
  (`/etc/gitattributes` тАФ e.g. Git-for-Windows' `astextplain` on
  `*.pdf`/`*.docx`) and global/XDG attributes files; only the PR's committed
  `.gitattributes` (via `attr.tree=<head>`) apply, matching GitHub.
- **Argv can't be an option.** `mb` and `head` are hex-validated before any
  git call and passed after `--end-of-options`, so an API-sourced value can
  never be parsed as a git flag.
- **Pinned to the evaluated head.** The fallback refuses to diff any head
  other than the one the view evidence recorded, so the verdict, the diff,
  and the eventual `--match-head-commit` merge all name the same SHA.
- **Fail closed.** No partial or empty diffs: any nonzero git exit, a
  merge base equal to head, or empty output aborts the run with no diff
  evidence written.
- **Narrow trigger.** Only the 406 too-large rejection (status *and* body)
  routes here, so the fallback cannot be steered by inducing other failures.
- **Auditable.** The diff artifact stamps `method`/`head`/`merge_base`, so
  which path produced the evidence and which commits it spans are
  reconstructable from state alone.

## Acceptance

- Unit: too-large detection (406 vs other errors), pulls/compare payload
  parsing, head-pin mismatch refusal тАФ table-driven, no network.
- A/B: fallback output byte-identical to `gh pr diff` on an under-cap PR.
- E2E: `gate gate` on an over-cap PR (workbench#51) reaches a verdict.
